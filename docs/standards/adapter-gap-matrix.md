# Adapter Gap Matrix

**Status:** active
**Scope:** provider coverage and normalization readiness before component work depends on it
**Purpose:** make provider coverage and normalization risk visible before component work depends on it

Status key:

- `supported`
- `partial`
- `unsupported`
- `unknown`
- `aggregate` (chart consumes pre-aggregated input; no per-provider adapter dependency at the chart layer)

## Required workflow

Before starting a component:

1. Identify the minimum normalized fields it requires.
2. Update this matrix.
3. Call out provider-specific lossy mappings in the component spec.
4. Cross-check `/Volumes/WQ/ref_code/INDEX.md` for relevant prior art only if that prior art covers the component or its underlying primitives.

## Current known gaps to resolve next

- Wyscout lineup/team-sheet support now exists, but it is explicitly tied to the public research dataset seam plus player/team lookups rather than a claimed official API parity surface
- FotMob direct JSON is currently Turnstile-protected; any future adapter would need to be framed as scrape-backed and experimental
- explicit product contract for average player position and pass-network thresholds
- schema and adapter path for any freeze-frame or player-position-driven chart
- formal decision on which providers can support xG-driven vs non-xG-driven variants by component

## StatsBomb event-type coverage audit

Cross-referenced against StatsBomb Open Data Events v4.0.0 spec (36 event type IDs) using sampled frequencies from 40 matches of `/Volumes/WQ/ref_code/statsbomb-open-data`. Frequencies below are raw sample counts for relative sizing, not per-match rates.

**Mapped by `fromStatsBomb.events()`** (13 StatsBomb types → 15 canonical outputs):

| type.id | StatsBomb name | Canonical kind                                                         |
| ------- | -------------- | ---------------------------------------------------------------------- |
| 2       | Ball Recovery  | `recovery`                                                             |
| 4       | Duel           | `tackle` if `duel.type.name==="Tackle"` else `duel`                    |
| 9       | Clearance      | `clearance`                                                            |
| 10      | Interception   | `interception`                                                         |
| 14      | Dribble        | `take-on`                                                              |
| 16      | Shot           | `shot`                                                                 |
| 17      | Pressure       | `pressure` (with `counterpress` first-class field)                     |
| 19      | Substitution   | `substitution`                                                         |
| 22      | Foul Committed | `foul-committed` (+ optional `card` when `foul_committed.card` is set) |
| 23      | Goal Keeper    | `goalkeeper`                                                           |
| 24      | Bad Behaviour  | `card` (only when `bad_behaviour.card` is set; out-of-play bookings)   |
| 30      | Pass           | `pass`                                                                 |

**Mapped by non-`events()` product surfaces** (consumed from the same stream):

| type.id | StatsBomb name | Surface                          |
| ------- | -------------- | -------------------------------- |
| 35      | Starting XI    | `matchLineups()`, `formations()` |
| 36      | Tactical Shift | `matchLineups()`, `formations()` |

**Intentionally filtered out of product shots** (reachable via `events()`-equivalent paths, dropped in `projectShots`): 20 Own Goal Against, 25 Own Goal For.

**Intentionally unmapped lifecycle / broadcast metadata** (would be noise in any product stream): 5 Camera On (deprecated), 8 Offside (outcomes surface on owning Pass), 18 Half Start, 26 Player On, 27 Player Off, 29 Camera Off (deprecated), 34 Half End, 40 Injury Stoppage, 41 Referee Ball-Drop.

**Peer-parity gaps (closed):** Ball Recovery (2) → `recovery`, Dribble (14) → `take-on`, and Bad Behaviour (24) → `card` are now emitted by `fromStatsBomb.events()`. The StatsBomb adapter is at parity with Opta / WhoScored / Wyscout for `recovery` and `take-on`, and handles out-of-play bookings via the shared `mapCard`.

**Design-decision gaps** — no canonical `EVENT_KINDS` slot today, so unmapping is defensible until a chart needs them:

### Remaining follow-ups

Schema-level decisions deferred: `block`, `error`, `foul-won`, richer duel variants. File as a schema RFC when a chart actually needs them. (`pressure` graduated from this list once `fromStatsBomb` landed `mapPressure` with a first-class `counterpress` field, backing press-map components.)
